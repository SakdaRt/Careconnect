# 1.3 ขอบเขตของโครงงาน — 3 เวอร์ชัน

---

## เวอร์ชัน A: เวอร์ชันหลัก (พร้อมใส่รายงาน ~850 คำ)

### 1.3 ขอบเขตของโครงงาน

โครงงานนี้เป็นการพัฒนาระบบเว็บแอปพลิเคชันต้นแบบ (Prototype) สำหรับบริการจับคู่ผู้ว่าจ้างกับผู้ดูแลผู้สูงอายุในประเทศไทย ในรูปแบบ Two-sided Marketplace โดย deploy บนเซิร์ฟเวอร์ของมหาวิทยาลัยที่มีทรัพยากรจำกัด ระบบมุ่งเน้นการสาธิตแนวคิดและความเป็นไปได้ทางเทคนิคมากกว่าการใช้งานเชิงพาณิชย์ ขอบเขตการพัฒนามีรายละเอียดดังนี้

**ขอบเขตด้านผู้ใช้งาน**

1) ระบบรองรับผู้ใช้ 3 บทบาท ได้แก่ ผู้ว่าจ้าง (Hirer) ผู้ดูแล (Caregiver) และผู้ดูแลระบบ (Admin)
2) ผู้ว่าจ้างและผู้ดูแลสามารถสมัครสมาชิกได้ 2 ช่องทาง คือ Guest (อีเมล) และ Member (เบอร์โทรศัพท์) รวมถึงเข้าสู่ระบบผ่าน Google OAuth ซึ่งเป็นบริการยืนยันตัวตนจากบุคคลที่สาม (Third-party Authentication)
3) ผู้ใช้แต่ละคนมีระดับความน่าเชื่อถือ (Trust Level) 4 ระดับ (L0–L3) ซึ่งคำนวณจากการยืนยันตัวตน ประวัติการใช้งาน และคะแนนรีวิว โดยส่งผลต่อสิทธิ์การเข้าถึงฟังก์ชันต่าง ๆ ในระบบ
4) ผู้ดูแลระบบมีหน้าจอเฉพาะสำหรับบริหารจัดการผู้ใช้ ตรวจสอบเอกสาร KYC จัดการข้อพิพาท และดูแลธุรกรรมทางการเงินของแพลตฟอร์ม

**ขอบเขตด้านฟังก์ชันหลัก**

5) ระบบยืนยันตัวตน: รองรับการสมัครด้วยอีเมลหรือเบอร์โทรศัพท์ การเข้าสู่ระบบผ่าน Google OAuth การยืนยัน OTP ผ่านอีเมลและ SMS (ผ่านบริการ SMSOK) การลืมรหัสผ่าน และการจัดการโปรไฟล์พร้อมอัปโหลดรูปภาพ
6) ระบบจัดการงาน: ผู้ว่าจ้างสร้างงานผ่านแบบฟอร์มแบบ 5 ขั้นตอน (Guided Wizard) ระบบคำนวณระดับความเสี่ยงของงานโดยอัตโนมัติ ผู้ดูแลค้นหาและรับงานจาก Job Feed รองรับการมอบหมายงานโดยตรง การยกเลิกงาน และการประกาศงานซ้ำได้สูงสุด 3 ครั้ง
7) ระบบลงเวลาปฏิบัติงาน: ผู้ดูแลลงเวลาเข้า-ออกงาน (Check-in/Check-out) ผ่าน Geolocation API ของเบราว์เซอร์ โดยบันทึกพิกัดและค่าความแม่นยำลงฐานข้อมูล รองรับการส่งงานก่อนเวลาโดยต้องได้รับอนุมัติจากผู้ว่าจ้าง
8) ระบบการเงิน: ใช้รูปแบบบัญชีคู่ (Double-entry Ledger) ประกอบด้วยกระเป๋าเงิน 5 ประเภท (ผู้ว่าจ้าง ผู้ดูแล Escrow แพลตฟอร์ม และทดแทน) รองรับการเติมเงิน ระบบพักเงิน (Escrow) ค่าธรรมเนียม 10% ที่หักจากค่าจ้าง เงินมัดจำผู้ว่าจ้างแบบขั้นบันได และค่าปรับกรณียกเลิกงาน
9) ระบบสื่อสาร: แชทแบบเรียลไทม์ผ่าน Socket.IO ระหว่างผู้ว่าจ้างและผู้ดูแล รองรับสถานะการพิมพ์ (Typing Indicator) และการอ่านข้อความ (Read Receipts)
10) ระบบแจ้งเตือน: แจ้งเตือนภายในแอปพลิเคชันแบบเรียลไทม์ผ่าน Socket.IO โดยมี 8 เหตุการณ์ที่สร้างการแจ้งเตือน เช่น มีผู้รับงาน ลงเวลาเข้างาน งานเสร็จสมบูรณ์ เป็นต้น ระบบรองรับโครงสร้างการแจ้งเตือนผ่านอีเมลและ Push Notification (PWA) แต่ยังอยู่ในระดับต้นแบบ
11) ระบบรีวิว: ผู้ว่าจ้างให้คะแนน 1–5 ดาวพร้อมความคิดเห็นหลังงานเสร็จ คะแนนส่งผลต่อ Trust Score ของผู้ดูแลโดยอัตโนมัติ
12) ระบบข้อพิพาทและร้องเรียน: ผู้ใช้เปิดข้อพิพาทพร้อมสนทนาโต้ตอบ ผู้ดูแลระบบจัดการ Settlement ทางการเงินได้ นอกจากนี้ยังมีระบบร้องเรียนทั่วไปพร้อมไฟล์แนบ
13) หน้าผู้ดูแลระบบ: ประกอบด้วย Dashboard จัดการผู้ใช้ ตรวจสอบ KYC จัดการงาน จัดการข้อพิพาท รายงานการเงินพร้อมส่งออก CSV และรายงานสรุป

**ขอบเขตด้านการเชื่อมต่อบริการภายนอก**

14) การชำระเงินเชื่อมต่อกับ Stripe ในโหมด Sandbox (Test Mode) สำหรับการเติมเงินผ่าน Checkout Session ไม่มีการใช้เงินจริง การถอนเงินเป็นกระบวนการ Manual ผ่านผู้ดูแลระบบ
15) การส่ง SMS OTP เชื่อมต่อกับ SMSOK สำหรับส่งรหัสยืนยันเบอร์โทรศัพท์
16) การเข้าสู่ระบบเชื่อมต่อกับ Google OAuth 2.0 (Authorization Code Flow)
17) การยืนยันตัวตน (KYC) เป็นการจำลองการทำงาน (Mockup) โดยระบบอนุมัติทันทีหลังส่งเอกสาร หน้าจอรองรับการอัปโหลดเอกสาร 3 รายการ (บัตรประชาชนด้านหน้า-หลัง และภาพถ่ายเซลฟี) แต่ยังไม่เชื่อมต่อผู้ให้บริการ KYC จริง ผู้ให้บริการที่วางแผนไว้คือ SCBX แต่ยังไม่ได้ดำเนินการเชื่อมต่อ
18) การส่งอีเมลรองรับ SMTP แต่ค่าเริ่มต้นใช้โหมดจำลอง Push Notification มีโครงสร้าง PWA พร้อมใช้งาน แต่ยังอยู่ในระดับต้นแบบ

**ขอบเขตด้านการออกแบบและอุปกรณ์ที่รองรับ**

19) ออกแบบแบบ Mobile-first ด้วย Tailwind CSS มี Bottom Navigation Bar สำหรับมือถือ รองรับ iOS Safe Area และ Progressive Web App (PWA)
20) แสดงผลเป็นภาษาไทย ใช้แบบอักษร Sarabun และ Noto Sans Thai ตรวจสอบความสามารถในการเข้าถึงตามแนวทาง WCAG 2.1 ระดับ AA

**ขอบเขตด้านข้อจำกัดของระบบ**

21) ระบบเป็นต้นแบบเพื่อการศึกษา deploy บนเซิร์ฟเวอร์มหาวิทยาลัยที่มีทรัพยากรจำกัด ไม่มีผู้ใช้จริงเชิงพาณิชย์
22) ใช้ Docker Compose แบบ Single Instance ไม่มีระบบ Caching, Load Balancing หรือ Horizontal Scaling
23) การชำระเงินผ่าน Stripe อยู่ในโหมด Sandbox (Test Mode) ไม่มีธุรกรรมเงินจริง
24) การยืนยันตัวตน KYC เป็นการจำลองการทำงาน (Mockup) ไม่มีการตรวจสอบความถูกต้องของเอกสาร
25) ไฟล์ที่อัปโหลดเก็บบน Local Disk ของเซิร์ฟเวอร์ ไม่ได้ใช้ Cloud Storage
26) ระบบตรวจจับการทุจริตทาง GPS มีโครงสร้างฐานข้อมูลรองรับ แต่ยังไม่พัฒนา Logic การตรวจจับขั้นสูง

## เวอร์ชัน B: กระชับมาก (~350 คำ)

### 1.3 ขอบเขตของโครงงาน

โครงงานนี้พัฒนาเว็บแอปพลิเคชันต้นแบบสำหรับบริการจับคู่ผู้ว่าจ้างกับผู้ดูแลผู้สูงอายุ ในรูปแบบ Two-sided Marketplace โดย deploy บนเซิร์ฟเวอร์มหาวิทยาลัยที่มีทรัพยากรจำกัด มีขอบเขตดังนี้

**ด้านผู้ใช้งาน** — รองรับ 3 บทบาท ได้แก่ ผู้ว่าจ้าง ผู้ดูแล และผู้ดูแลระบบ สมัครสมาชิกได้ด้วยอีเมล เบอร์โทรศัพท์ หรือ Google OAuth มีระบบ Trust Level 4 ระดับที่คำนวณจากการยืนยันตัวตนและประวัติการใช้งาน

**ด้านฟังก์ชันหลัก** — ครอบคลุมระบบยืนยันตัวตน (OTP ผ่าน SMSOK, Google OAuth), ระบบจัดการงาน (สร้าง-เผยแพร่-รับงาน-ลงเวลาด้วย GPS-ส่งงาน), ระบบการเงินแบบบัญชีคู่ (Escrow, ค่าธรรมเนียม 10%, เงินมัดจำ, ค่าปรับ), แชทเรียลไทม์ผ่าน Socket.IO, ระบบแจ้งเตือนภายในแอป, ระบบรีวิว 1–5 ดาว, ระบบข้อพิพาทพร้อม Settlement ทางการเงิน และหน้าผู้ดูแลระบบสำหรับจัดการผู้ใช้ KYC ข้อพิพาท และรายงานการเงิน

**ด้านบริการภายนอก** — การชำระเงินเชื่อมต่อ Stripe ในโหมด Sandbox ไม่มีธุรกรรมเงินจริง SMS OTP ส่งจริงผ่าน SMSOK เข้าสู่ระบบผ่าน Google OAuth (Third-party Authentication) การยืนยันตัวตน KYC เป็นการจำลองการทำงาน (Mockup) ยังไม่เชื่อมต่อผู้ให้บริการจริง โดย SCBX เป็นผู้ให้บริการที่วางแผนไว้สำหรับอนาคต

**ด้านการออกแบบ** — ออกแบบ Mobile-first ด้วย Tailwind CSS รองรับ PWA แสดงผลภาษาไทย ตรวจสอบ Accessibility ตามแนวทาง WCAG 2.1 ระดับ AA

**ด้านข้อจำกัด** — เป็นระบบต้นแบบเพื่อการศึกษา ใช้ Docker Compose แบบ Single Instance บนเซิร์ฟเวอร์มหาวิทยาลัย ไม่มี Caching หรือ Scaling การชำระเงินเป็น Sandbox KYC เป็น Mockup ไฟล์เก็บบน Local Disk และระบบตรวจจับทุจริต GPS มีโครงสร้างพร้อมแต่ยังไม่พัฒนา Logic ขั้นสูง

## เวอร์ชัน C: แบ่งหัวข้อย่อยชัดเจน อ่านง่ายที่สุด (~800 คำ)

### 1.3 ขอบเขตของโครงงาน

โครงงานนี้เป็นการพัฒนาเว็บแอปพลิเคชันต้นแบบ (Prototype) สำหรับบริการจับคู่ผู้ว่าจ้างกับผู้ดูแลผู้สูงอายุในประเทศไทย ในรูปแบบ Two-sided Marketplace โดย deploy บนเซิร์ฟเวอร์มหาวิทยาลัยที่มีทรัพยากรจำกัด

#### 1.3.1 ขอบเขตด้านผู้ใช้งาน

- ระบบรองรับผู้ใช้ 3 บทบาท ได้แก่ ผู้ว่าจ้าง (Hirer) ผู้ดูแล (Caregiver) และผู้ดูแลระบบ (Admin)
- สมัครสมาชิกได้ 2 ช่องทาง คือ อีเมล (Guest) และเบอร์โทรศัพท์ (Member) รวมถึง Google OAuth
- มีระบบระดับความน่าเชื่อถือ (Trust Level) 4 ระดับ คำนวณจากการยืนยันตัวตน ประวัติการใช้งาน และคะแนนรีวิว
- ผู้ดูแลระบบมีหน้าจอแยกสำหรับบริหารจัดการผู้ใช้ ตรวจสอบ KYC จัดการข้อพิพาท และดูแลการเงิน

#### 1.3.2 ขอบเขตด้านฟังก์ชันหลัก

**ระบบยืนยันตัวตนและจัดการบัญชี**
- สมัครด้วยอีเมล เบอร์โทร หรือ Google OAuth (Third-party Authentication)
- ยืนยัน OTP ผ่านอีเมลและ SMS (SMSOK)
- จัดการโปรไฟล์ อัปโหลดรูปภาพ ลืม/เปลี่ยนรหัสผ่าน

**ระบบจัดการงาน**
- สร้างงานผ่านแบบฟอร์ม 5 ขั้นตอน (Guided Wizard)
- คำนวณระดับความเสี่ยงอัตโนมัติ (low_risk / high_risk)
- ค้นหาและรับงานจาก Job Feed พร้อมระบบกรอง
- มอบหมายงานโดยตรง (Direct Assignment)
- ยกเลิกงาน และประกาศซ้ำได้สูงสุด 3 ครั้ง

**ระบบลงเวลาปฏิบัติงาน**
- Check-in/Check-out ผ่าน Geolocation API ของเบราว์เซอร์ บันทึกพิกัดลงฐานข้อมูล
- ส่งงานก่อนเวลาได้โดยต้องได้รับอนุมัติจากผู้ว่าจ้าง

**ระบบการเงิน**
- บัญชีคู่ (Double-entry Ledger) พร้อมกระเป๋าเงิน 5 ประเภท
- เติมเงินผ่าน Stripe Checkout, ระบบพักเงิน (Escrow)
- ค่าธรรมเนียมแพลตฟอร์ม 10% หักจากค่าจ้าง
- เงินมัดจำแบบขั้นบันได และค่าปรับกรณียกเลิกงาน

**ระบบสื่อสารและแจ้งเตือน**
- แชทเรียลไทม์ผ่าน Socket.IO พร้อม Typing Indicator และ Read Receipts
- แจ้งเตือนภายในแอปแบบเรียลไทม์ 8 เหตุการณ์
- โครงสร้างรองรับอีเมลและ Push Notification (PWA) ในระดับต้นแบบ

**ระบบรีวิวและข้อพิพาท**
- รีวิว 1–5 ดาวหลังงานเสร็จ ส่งผลต่อ Trust Score อัตโนมัติ
- เปิดข้อพิพาทพร้อมสนทนา ผู้ดูแลระบบจัดการ Settlement ทางการเงิน
- ระบบร้องเรียนทั่วไปพร้อมไฟล์แนบ

**หน้าผู้ดูแลระบบ**
- Dashboard, จัดการผู้ใช้, ตรวจสอบ KYC, จัดการงาน, จัดการข้อพิพาท
- รายงานการเงิน (ธุรกรรม การถอนเงิน Settlement) พร้อมส่งออก CSV

#### 1.3.3 ขอบเขตด้านบริการภายนอก

| บริการ | ผู้ให้บริการ | สถานะการเชื่อมต่อ |
|--------|------------|------------------|
| การชำระเงิน | Stripe | Sandbox (Test Mode) — ไม่มีธุรกรรมเงินจริง |
| SMS OTP | SMSOK | เชื่อมต่อใช้งานจริง |
| เข้าสู่ระบบ | Google OAuth 2.0 | เชื่อมต่อใช้งานจริง |
| ยืนยันตัวตน (KYC) | Mockup (SCBX วางแผนไว้) | จำลองการทำงาน — อนุมัติทันที ยังไม่เชื่อมจริง |
| อีเมล | SMTP (nodemailer) | รองรับโครงสร้าง ค่าเริ่มต้นเป็นโหมดจำลอง |
| Push Notification | Web Push API (PWA) | มีโครงสร้างพร้อม อยู่ในระดับต้นแบบ |

#### 1.3.4 ขอบเขตด้านการออกแบบ

- ออกแบบ Mobile-first ด้วย Tailwind CSS มี Bottom Navigation สำหรับมือถือ
- รองรับ iOS Safe Area และ Progressive Web App (PWA)
- แสดงผลภาษาไทย ใช้แบบอักษร Sarabun และ Noto Sans Thai
- ตรวจสอบ Accessibility ตามแนวทาง WCAG 2.1 ระดับ AA

#### 1.3.5 ขอบเขตด้านข้อจำกัด

- เป็นระบบต้นแบบเพื่อการศึกษา ไม่มีผู้ใช้จริงเชิงพาณิชย์
- Deploy ด้วย Docker Compose แบบ Single Instance บนเซิร์ฟเวอร์มหาวิทยาลัยที่มีทรัพยากรจำกัด
- ไม่มีระบบ Caching (Redis), Load Balancing หรือ Horizontal Scaling
- การชำระเงินผ่าน Stripe เป็นโหมด Sandbox ไม่มีธุรกรรมเงินจริง
- การยืนยันตัวตน (KYC) เป็นการจำลอง (Mockup) ไม่ตรวจสอบเอกสารจริง โดย SCBX เป็นผู้ให้บริการที่วางแผนรองรับในอนาคต
- ไฟล์ที่อัปโหลดเก็บบน Local Disk ไม่ใช้ Cloud Storage
- ระบบตรวจจับทุจริตทาง GPS มีโครงสร้างฐานข้อมูลพร้อม แต่ยังไม่พัฒนา Logic ขั้นสูง
