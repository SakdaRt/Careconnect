# บทที่ 2 เครื่องมือที่เกี่ยวข้อง

---

## 2.1 ทฤษฎีที่เกี่ยวข้อง

การพัฒนาแพลตฟอร์มบริการดูแลผู้สูงอายุอย่างแคร์คอนเน็กต์ จำเป็นต้องอาศัยหลักการออกแบบระบบที่หลากหลาย เพื่อให้ระบบสามารถตอบสนองความต้องการของผู้ใช้งานได้อย่างมีประสิทธิภาพ ทฤษฎีและหลักการออกแบบที่เกี่ยวข้องสามารถสรุปได้ดังนี้

### 2.1.1 Chat Application System Design

ระบบแชทที่มีประสิทธิภาพสูงเป็นระบบที่ต้องการการออกแบบอย่างรอบคอบ เพื่อให้สอดคล้องกับความต้องการของผู้ใช้งานทั้งในเชิงฟังก์ชันและเชิงไม่ฟังก์ชัน ระบบที่นำเสนอนี้ประกอบด้วยฟังก์ชันการทำงานที่สำคัญ เช่น การรองรับการสนทนาแบบหนึ่งต่อหนึ่งภายในบริบทของงาน (Thread-based Chat) การยืนยันการส่งข้อความ การจัดเก็บประวัติการสนทนา รวมถึงการแจ้งเตือนผู้ใช้เมื่อมีข้อความใหม่เข้ามา ฟังก์ชันเหล่านี้มีความสำคัญอย่างยิ่งต่อประสบการณ์การใช้งาน เพื่อให้ผู้ว่าจ้างและผู้ดูแลสามารถประสานงานกันได้อย่างมีประสิทธิภาพ

รูปที่ 2.1 หลักการทำงานของระบบการแชททั่วไป

นอกจากฟังก์ชันการทำงานพื้นฐานแล้ว ยังมีความต้องการเชิงไม่ฟังก์ชันที่จำเป็นต้องพิจารณา เช่น ความหน่วงต่ำ (Low Latency) ความสอดคล้องของข้อมูล (Data Consistency) ความพร้อมใช้งาน (Availability) และความปลอดภัย โดยเฉพาะอย่างยิ่งการรักษาความปลอดภัยของข้อมูลการสนทนาระหว่างผู้ว่าจ้างและผู้ดูแล ซึ่งอาจมีข้อมูลส่วนบุคคลของผู้รับการดูแล นอกจากนี้ระบบจะต้องสามารถรองรับการขยายตัวของจำนวนผู้ใช้และข้อความที่เพิ่มขึ้นได้อย่างมีประสิทธิภาพ

ในการออกแบบระบบระดับสูง ระบบแชทจะต้องมีองค์ประกอบหลักหลายประการ เช่น การสื่อสารระหว่างผู้ใช้ผ่านเซิร์ฟเวอร์แชท การจัดการการเชื่อมต่อระหว่างผู้ใช้และเซิร์ฟเวอร์ รวมถึงการเก็บรักษาข้อความเมื่อผู้รับออฟไลน์ การใช้โปรโตคอล WebSocket เป็นทางเลือกที่เหมาะสม เนื่องจากรองรับการสื่อสารแบบสองทาง (Bi-directional) แบบเรียลไทม์ ลดความหน่วงและให้ประสิทธิภาพที่ดีกว่าเทคนิคอื่น ๆ เช่น Polling หรือ Long Polling ในแคร์คอนเน็กต์ใช้ Socket.IO ซึ่งเป็นไลบรารีที่สร้างบน WebSocket เพื่อให้การเชื่อมต่อมีความเสถียรและรองรับ Fallback อัตโนมัติ

ระบบที่ออกแบบขึ้นควรแบ่งออกเป็นบริการที่ไม่มีสถานะ (Stateless Services) และบริการที่มีสถานะ (Stateful Services) บริการที่ไม่มีสถานะจะใช้สำหรับการดำเนินการที่ไม่จำเป็นต้องมีการเชื่อมต่อที่ยาวนาน เช่น การลงทะเบียนและการเข้าสู่ระบบ ในขณะที่บริการที่มีสถานะจะจัดการกับการสนทนาแบบเรียลไทม์ที่ต้องการการเชื่อมต่อระยะยาว

ในส่วนของการจัดเก็บข้อมูล ระบบต้องรองรับข้อมูลสองประเภท ได้แก่ ข้อมูลทั่วไปของผู้ใช้ เช่น โปรไฟล์และการตั้งค่า ซึ่งสามารถจัดเก็บในฐานข้อมูลเชิงสัมพันธ์ และข้อมูลประวัติการสนทนาที่ต้องการการจัดเก็บที่สามารถขยายขนาดได้อย่างมีประสิทธิภาพ ในแคร์คอนเน็กต์ใช้ PostgreSQL ในการจัดเก็บข้อมูลทั้งสองประเภท โดยออกแบบตาราง chat_threads และ chat_messages เพื่อรองรับระบบแชทแบบ Thread-based ที่ผูกกับงานแต่ละงาน

### 2.1.2 Filter Search

Search Filters เป็นเครื่องมือที่ช่วยปรับแต่งผลการค้นหาและจำกัดตัวเลือก โดยให้ผู้ใช้สามารถค้นหาผลลัพธ์ที่ตรงตามคุณลักษณะที่ต้องการได้ การใช้ฟิลเตอร์ช่วยให้ผู้ใช้นำทางระบบได้อย่างราบรื่นและค้นพบสิ่งที่ตรงความต้องการได้แม่นยำขึ้น ฟิลเตอร์การค้นหานี้เป็นส่วนหนึ่งของฟีเจอร์การเรียงลำดับอย่างง่ายและการค้นหาแบบ Faceted Search ที่ซับซ้อนยิ่งขึ้น

2.1.2.1 ความสำคัญของ Search Filters

ในแพลตฟอร์มบริการดูแลผู้สูงอายุ ผู้ว่าจ้างอาจไม่แน่ใจว่าต้องการผู้ดูแลที่มีคุณสมบัติอย่างไร ฟิลเตอร์จึงช่วยให้ค้นหาผู้ดูแลที่ตรงตามเงื่อนไขที่ระบุไว้ได้ง่ายขึ้น และยังเปิดโอกาสให้ผู้ว่าจ้างพบผู้ดูแลที่เหมาะสมที่ไม่เคยพิจารณามาก่อน ซึ่งช่วยลดความกังวลในการเลือกผู้ดูแลจากรายชื่อจำนวนมาก นอกจากนี้ฟิลเตอร์ที่จัดการได้อย่างถูกต้องยังเพิ่มประสบการณ์ที่ดีและประสิทธิภาพของแพลตฟอร์ม

2.1.2.2 การทำงานของ Search Filters

ฟิลเตอร์การค้นหาที่มีประสิทธิภาพอาศัยการจัดหมวดหมู่ข้อมูล (Tagging) ที่มีความแม่นยำและสอดคล้องกัน ในแคร์คอนเน็กต์ ระบบค้นหาผู้ดูแลใช้ฟิลเตอร์หลายมิติ ได้แก่ ทักษะและความเชี่ยวชาญ (เช่น ดูแลผู้ป่วยอัลไซเมอร์ กายภาพบำบัด) พื้นที่ให้บริการ ช่วงราคา ระดับความน่าเชื่อถือ (Trust Level) คะแนนรีวิว และตารางเวลาว่าง การใช้ฟิลเตอร์แบบ Faceted Search ช่วยให้ผู้ว่าจ้างสามารถจำกัดผลการค้นหาได้แม่นยำยิ่งขึ้น

2.1.2.3 แนวทางปฏิบัติที่ดีที่สุดของ Search Filters

1) เพิ่มฟิลเตอร์เฉพาะตามประเภทการดูแลเพื่อให้เหมาะกับความต้องการแต่ละกรณี
2) ใช้ฟิลเตอร์ตามระดับความน่าเชื่อถือ (Trust Level) สำหรับผู้ว่าจ้างที่ต้องการผู้ดูแลที่ผ่านการยืนยันตัวตนแล้ว
3) แสดงจำนวนผลลัพธ์ที่ตรงตามเงื่อนไขเพื่อช่วยให้ผู้ใช้ตัดสินใจปรับฟิลเตอร์
4) หลีกเลี่ยงการแสดงผล "ไม่พบผู้ดูแล" โดยสามารถแนะนำผู้ดูแลที่มีคุณสมบัติใกล้เคียงแทน

### 2.1.3 QR Code Payment

การชำระเงินด้วย QR Code เป็นวิธีการที่ได้รับความนิยมในปัจจุบัน เนื่องจากความสะดวกสบายและความรวดเร็ว ระบบนี้จะสร้าง QR Code ที่มีข้อมูลการชำระเงินฝังอยู่ เช่น ยอดเงิน หมายเลขบัญชี หรือข้อมูลเพิ่มเติมอื่น ๆ ผู้ใช้สามารถสแกน QR Code ผ่านแอปพลิเคชันการชำระเงิน เช่น แอปธนาคาร หรือกระเป๋าเงินดิจิทัล จากนั้นข้อมูลจะถูกส่งไปยังเซิร์ฟเวอร์เพื่อดำเนินการชำระเงิน

ในแคร์คอนเน็กต์ ระบบ QR Code Payment ถูกนำมาใช้ในการเติมเงินเข้ากระเป๋าเงิน (Wallet Top-up) โดยเมื่อผู้ว่าจ้างต้องการเติมเงิน ระบบจะสร้าง QR Code ผ่าน Payment Provider แสดงใน Popup ให้ผู้ใช้สแกนผ่านแอปธนาคาร เมื่อชำระเงินแล้วผู้ใช้กดยืนยัน ระบบจะตรวจสอบสถานะการชำระเงินกับ Payment Provider และอัปเดตยอดเงินในกระเป๋าเงินโดยอัตโนมัติ

การออกแบบระบบชำระเงินด้วย QR Code ต้องคำนึงถึง:
1) ความปลอดภัย เช่น การเข้ารหัสข้อมูลภายใน QR Code การกำหนดอายุการใช้งานของ QR Code และการป้องกันการสแกนซ้ำ
2) ความเข้ากันได้กับหลายแพลตฟอร์ม เช่น การรองรับแอปธนาคารและกระเป๋าเงินดิจิทัลที่แตกต่างกัน
3) การจัดการสถานะ (State Management) เช่น การตรวจสอบว่าการชำระเงินสำเร็จหรือไม่ การจัดการกรณี Timeout และการ Retry

### 2.1.4 Escrow Payment System Design

ระบบ Escrow เป็นกลไกการชำระเงินที่มีตัวกลางทำหน้าที่ถือเงินไว้ชั่วคราว จนกว่าทั้งสองฝ่ายจะปฏิบัติตามเงื่อนไขที่ตกลงกันไว้ครบถ้วน จึงจะปล่อยเงินให้กับผู้รับ ระบบ Escrow มีความสำคัญอย่างยิ่งในแพลตฟอร์มที่เชื่อมต่อผู้ใช้งานสองกลุ่มที่ไม่รู้จักกัน เพราะช่วยลดความเสี่ยงสำหรับทั้งผู้ซื้อและผู้ขาย

รูปที่ 2.2 หลักการทำงานของระบบ Escrow Payment

2.1.4.1 วัตถุประสงค์

วัตถุประสงค์หลักคือการออกแบบระบบ Escrow ที่เหมาะสมกับแพลตฟอร์มบริการดูแลผู้สูงอายุ โดยรับประกันว่าผู้ดูแลจะได้รับค่าตอบแทนเมื่อปฏิบัติงานเสร็จสิ้น และผู้ว่าจ้างจะได้รับบริการตามที่ตกลงไว้

2.1.4.2 ขั้นตอนการทำงานของ Escrow

ในแคร์คอนเน็กต์ ระบบ Escrow ทำงานตามขั้นตอนดังนี้:
1) การกันเงิน (Hold): เมื่อผู้ดูแลตอบรับงาน ระบบจะหักเงินจากกระเป๋าเงินของผู้ว่าจ้างและเก็บไว้ในบัญชี Escrow ของระบบ
2) การปฏิบัติงาน: ผู้ดูแลเดินทางไปยังสถานที่และทำการ Check-in ด้วย GPS เพื่อยืนยันการเริ่มงาน
3) การปล่อยเงิน (Release): เมื่อผู้ดูแลทำการ Check-out และงานเสร็จสมบูรณ์ ระบบจะปล่อยเงินจากบัญชี Escrow ไปยังกระเป๋าเงินของผู้ดูแลโดยอัตโนมัติ
4) การจัดการข้อพิพาท (Dispute): หากเกิดปัญหาระหว่างการทำงาน ทั้งสองฝ่ายสามารถเปิดข้อพิพาทได้ โดยเงินจะถูกระงับไว้จนกว่าผู้ดูแลระบบจะตัดสิน

2.1.4.3 การออกแบบระบบบัญชี

ระบบ Escrow ในแคร์คอนเน็กต์ใช้หลักการ Double-entry Ledger ในการบันทึกธุรกรรม โดยทุกการเคลื่อนไหวของเงินจะถูกบันทึกเป็นรายการเดบิตและเครดิตคู่กันเสมอ ทำให้สามารถตรวจสอบความถูกต้องของยอดเงินได้ตลอดเวลา ประเภทกระเป๋าเงินในระบบแบ่งออกเป็น:
1) User Wallet: กระเป๋าเงินส่วนตัวของผู้ใช้ (ทั้งผู้ว่าจ้างและผู้ดูแล)
2) Platform Escrow: บัญชีกลางที่ถือเงินไว้ระหว่างรอการปฏิบัติงาน
3) Platform Revenue: บัญชีรายได้ของแพลตฟอร์มจากค่าธรรมเนียม

### 2.1.5 Security of Web Application

ความปลอดภัยของเว็บแอปพลิเคชันเป็นหนึ่งในประเด็นสำคัญในการพัฒนาแอปพลิเคชันที่ผู้พัฒนาต้องให้ความสำคัญ ระบบที่ดีต้องป้องกันความเสี่ยงจากการโจมตี เช่น Cross-Site Scripting (XSS), SQL Injection, Cross-Site Request Forgery (CSRF) และการโจมตีแบบอื่น ๆ

แนวทางในการป้องกันรวมถึง:
1) การเข้ารหัสข้อมูล ทั้งที่เก็บในฐานข้อมูลและขณะส่งผ่านเครือข่าย (เช่น HTTPS) โดยในแคร์คอนเน็กต์ใช้ bcrypt ในการแฮชรหัสผ่านก่อนจัดเก็บ
2) การยืนยันตัวตนและการควบคุมสิทธิ์ (Authentication & Authorization) โดยใช้ JWT ในการจัดการ Session และ Role-based Access Control สำหรับการจำกัดสิทธิ์ตามบทบาทผู้ใช้ (Hirer, Caregiver, Admin)
3) การตรวจสอบความถูกต้องของข้อมูล (Input Validation) โดยใช้ Joi Schema ในการตรวจสอบข้อมูลที่รับเข้ามาทุก API endpoint เพื่อป้องกัน SQL Injection และข้อมูลที่ไม่ถูกต้อง
4) การทำงานร่วมกับเครื่องมือหรือเฟรมเวิร์คที่มีการอัปเดตด้านความปลอดภัยเป็นประจำ
5) การใช้ Parameterized Queries ในการเข้าถึงฐานข้อมูลผ่าน pg library เพื่อป้องกัน SQL Injection

### 2.1.6 OAuth 2.0

OAuth 2.0 เป็นมาตรฐานที่ใช้ในการยืนยันตัวตนและการอนุญาตเข้าถึงทรัพยากรบนระบบหรือบริการต่าง ๆ โดยไม่ต้องเปิดเผย Username และ Password แทนที่จะใช้ข้อมูลรับรองดังกล่าว OAuth 2.0 ใช้ Token ซึ่งสามารถควบคุมการเข้าถึงได้อย่างเจาะจง เช่น การกำหนดขอบเขต (Scope) ของข้อมูล ทำให้เมื่อ Token รั่วไหล ผลกระทบที่เกิดขึ้นสามารถควบคุมได้ในขอบเขตที่จำกัด

2.1.6.1 บทบาทของ OAuth 2.0

1) Authorization Code: ใช้สำหรับแอปพลิเคชันที่สามารถรักษาความปลอดภัยของข้อมูลได้ เช่น เว็บแอปพลิเคชัน โดยการร้องขอสิทธิ์ผ่าน Authorization Code และแลกเปลี่ยนกับ Access Token ในแคร์คอนเน็กต์ใช้วิธีนี้สำหรับการเข้าสู่ระบบผ่าน Google OAuth
2) Implicit: ใช้สำหรับ Mobile Apps และ Web Applications ที่มีความปลอดภัยน้อยกว่า โดย Token จะถูกส่งไปยังผู้ใช้ทันที ทำให้มีโอกาสที่ Token จะถูกเปิดเผยมากกว่า Implicit ไม่รองรับการใช้ Refresh Token เพื่อขอ Access Token ใหม่
3) Resource Owner Password Credentials: ผู้ใช้ส่ง Username และ Password ให้กับแอปพลิเคชันโดยตรง และแอปพลิเคชันจะใช้ข้อมูลนี้เพื่อขอ Access Token จากเซิร์ฟเวอร์ เหมาะกับกรณีที่ผู้ใช้เชื่อถือแอปพลิเคชันนั้นโดยตรง
4) Client Credentials: ใช้ในกรณีที่แอปพลิเคชันต้องการขอสิทธิ์ในการเข้าถึงข้อมูลของตนเอง โดยใช้ Client ID และ Client Secret เพื่อขอ Access Token จากเซิร์ฟเวอร์

2.1.6.2 การใช้ Access Token

เมื่อแอปพลิเคชันได้รับ Access Token แล้ว จะนำไปใช้เพื่อเข้าถึงบัญชีหรือข้อมูลของผู้ใช้ผ่าน API โดย API จะตรวจสอบว่า Access Token ถูกต้องและยังไม่หมดอายุ ในแคร์คอนเน็กต์ Access Token จะถูกส่งไปพร้อมกับทุก HTTP Request ในรูปแบบ Bearer Token ผ่าน Authorization Header

2.1.6.3 การใช้ Refresh Token

เมื่อนำ Access Token ที่หมดอายุไปใช้ API จะส่งข้อความกลับว่า Invalid Token และหากมี Refresh Token อยู่ แอปพลิเคชันสามารถใช้ Refresh Token เพื่อขอ Access Token ใหม่จาก Authorization Server ในแคร์คอนเน็กต์ ระบบรองรับ Refresh Token เพื่อให้ผู้ใช้ไม่ต้องเข้าสู่ระบบใหม่บ่อยครั้ง โดย Frontend จะจัดการการต่ออายุ Token โดยอัตโนมัติเมื่อ Access Token หมดอายุ

### 2.1.7 KYC และระบบยืนยันตัวตน (Know Your Customer)

KYC (Know Your Customer) เป็นกระบวนการยืนยันตัวตนของผู้ใช้งาน ซึ่งเป็นมาตรฐานที่ใช้กันอย่างแพร่หลายในอุตสาหกรรมการเงินและบริการออนไลน์ เพื่อป้องกันการฉ้อโกง การปลอมแปลงตัวตน และการฟอกเงิน กระบวนการ KYC โดยทั่วไปประกอบด้วยการตรวจสอบเอกสารระบุตัวตน (เช่น บัตรประชาชน หนังสือเดินทาง) การตรวจสอบที่อยู่ และการตรวจสอบความถูกต้องของข้อมูล

ในแคร์คอนเน็กต์ ระบบ KYC ถูกนำมาใช้เป็นส่วนหนึ่งของระบบระดับความน่าเชื่อถือ (Trust Level) ซึ่งแบ่งเป็น 4 ระดับ:
1) Level 0 (Unverified): ผู้ใช้ลงทะเบียนแล้วแต่ยังไม่ได้ยืนยันตัวตน สามารถเรียกดูข้อมูลได้แต่ไม่สามารถทำธุรกรรมได้
2) Level 1 (Email/Phone Verified): ผู้ใช้ยืนยันอีเมลหรือเบอร์โทรศัพท์ด้วย OTP สามารถสร้างงานหรือรับงานพื้นฐานได้
3) Level 2 (ID Verified): ผู้ใช้ส่งเอกสารยืนยันตัวตนและผ่านการตรวจสอบโดยผู้ดูแลระบบ สามารถเข้าถึงฟีเจอร์ทั้งหมดได้
4) Level 3 (Fully Verified): ผู้ดูแลที่ส่งใบรับรองวิชาชีพและผ่านการตรวจสอบครบถ้วน ได้รับป้ายยืนยันพิเศษบนโปรไฟล์

ระบบ Trust Level ช่วยสร้างความน่าเชื่อถือแบบเป็นขั้นตอน ทำให้ผู้ใช้ใหม่สามารถเริ่มใช้งานได้ทันทีและค่อย ๆ ปลดล็อกความสามารถเพิ่มเติมเมื่อยืนยันตัวตนมากขึ้น

### 2.1.8 Real-time Notification System Design

ระบบแจ้งเตือนแบบเรียลไทม์เป็นองค์ประกอบสำคัญของแพลตฟอร์มบริการที่ต้องการการตอบสนองอย่างรวดเร็ว เพื่อให้ผู้ใช้ได้รับข้อมูลที่สำคัญทันทีโดยไม่ต้องรีเฟรชหน้าเว็บ

2.1.8.1 ประเภทของการแจ้งเตือน

ในแพลตฟอร์มบริการดูแลผู้สูงอายุ การแจ้งเตือนสามารถแบ่งออกเป็นหลายประเภท:
1) การแจ้งเตือนเกี่ยวกับงาน เช่น มีผู้ดูแลรับงานแล้ว ผู้ดูแล Check-in แล้ว งานเสร็จสมบูรณ์
2) การแจ้งเตือนการเงิน เช่น เติมเงินสำเร็จ ได้รับค่าตอบแทน ถอนเงินสำเร็จ
3) การแจ้งเตือนแชท เช่น มีข้อความใหม่ในการสนทนา
4) การแจ้งเตือนระบบ เช่น การยืนยันตัวตนสำเร็จ การเปลี่ยนแปลง Trust Level

2.1.8.2 สถาปัตยกรรมระบบแจ้งเตือน

ระบบแจ้งเตือนที่มีประสิทธิภาพควรออกแบบด้วยสถาปัตยกรรม 2 ชั้น:
1) ชั้น Real-time: ใช้ WebSocket (Socket.IO) ในการส่งการแจ้งเตือนทันทีเมื่อผู้ใช้ออนไลน์ ทำให้ได้รับข้อมูลโดยไม่มีความหน่วง
2) ชั้น Fallback Polling: ใช้การสำรวจสถานะ (Polling) เป็นระยะเพื่อรองรับกรณีที่การเชื่อมต่อ WebSocket ขาดหาย หรือผู้ใช้กลับมาออนไลน์ใหม่

การออกแบบแบบ 2 ชั้นนี้ช่วยให้ระบบมีความเสถียรและไม่พลาดการแจ้งเตือนที่สำคัญ แม้ในสภาวะที่เครือข่ายไม่เสถียร

### 2.1.9 RESTful API Design

RESTful API (Representational State Transfer) เป็นรูปแบบสถาปัตยกรรมสำหรับการออกแบบ Web API ที่ใช้กันอย่างแพร่หลาย พัฒนาโดย Roy Fielding ในปี ค.ศ. 2000 RESTful API ยึดหลักการใช้ HTTP Methods (GET, POST, PUT, PATCH, DELETE) ในการกำหนดการกระทำต่อทรัพยากร (Resource) และใช้ URL ในการระบุทรัพยากรอย่างชัดเจน

2.1.9.1 หลักการสำคัญของ RESTful API

1) Stateless: ทุก Request ต้องมีข้อมูลที่จำเป็นครบถ้วนในตัว เซิร์ฟเวอร์ไม่เก็บสถานะของ Client ไว้ระหว่าง Request ในแคร์คอนเน็กต์ใช้ JWT Token แนบไปกับทุก Request เพื่อยืนยันตัวตนแทนการใช้ Session บนเซิร์ฟเวอร์
2) Resource-based: ทรัพยากรถูกระบุด้วย URI ที่ชัดเจน เช่น `/api/jobs`, `/api/wallet`, `/api/chat/threads`
3) Uniform Interface: ใช้ HTTP Methods มาตรฐานในการกระทำ เช่น GET สำหรับดึงข้อมูล POST สำหรับสร้างข้อมูล PUT/PATCH สำหรับแก้ไข DELETE สำหรับลบ
4) Layered System: ระบบแบ่งเป็นชั้น ๆ ได้ เช่น Routes → Controllers → Services → Database

2.1.9.2 การประยุกต์ใช้ใน CareConnect

ในแคร์คอนเน็กต์ Backend ทั้งหมดออกแบบเป็น RESTful API ด้วย Express.js โดยแบ่ง Routes ตามฟังก์ชันการทำงานเป็น 17 กลุ่ม ได้แก่ Auth, Jobs, Wallet, Chat, Notifications, Dispute, OTP, KYC, Reviews, Favorites, Caregiver Search, Care Recipients, Caregiver Documents, Complaints, Admin, Payments และ Webhooks ทุก Route ใช้ Middleware ในการตรวจสอบสิทธิ์ (requireAuth) และนโยบาย (requirePolicy) เพื่อควบคุมการเข้าถึงตามบทบาทและ Trust Level ของผู้ใช้

### 2.1.10 Geolocation API

Geolocation API เป็นมาตรฐานของ Web Browser ที่ช่วยให้เว็บแอปพลิเคชันสามารถขอตำแหน่งทางภูมิศาสตร์ของผู้ใช้ได้ โดยใช้ GPS, Wi-Fi หรือ Cell Tower ในการระบุตำแหน่ง ผู้ใช้ต้องอนุญาตการเข้าถึงตำแหน่งก่อนที่แอปพลิเคชันจะสามารถใช้งานได้

2.1.10.1 การใช้งานใน CareConnect

ในแคร์คอนเน็กต์ Geolocation API ถูกนำมาใช้ในระบบ Check-in/Check-out ของผู้ดูแล ซึ่งเป็นฟีเจอร์สำคัญในการยืนยันว่าผู้ดูแลเดินทางไปถึงสถานที่ปฏิบัติงานจริง:
1) Check-in: เมื่อผู้ดูแลถึงสถานที่ ระบบจะขอตำแหน่ง GPS ของผู้ดูแลผ่าน `navigator.geolocation.getCurrentPosition()` และบันทึกพิกัด (Latitude, Longitude) พร้อม Timestamp ลงในฐานข้อมูล
2) Check-out: เมื่อผู้ดูแลปฏิบัติงานเสร็จสิ้น ระบบจะบันทึกพิกัด GPS อีกครั้งพร้อมหลักฐานการทำงาน (Evidence Note) เพื่อใช้เป็นหลักฐานยืนยันการปฏิบัติงาน
3) การตรวจสอบระยะทาง: ระบบสามารถคำนวณระยะทางระหว่างตำแหน่งของผู้ดูแลกับตำแหน่งของสถานที่ปฏิบัติงานได้ เพื่อยืนยันว่าผู้ดูแลอยู่ในพื้นที่ที่ถูกต้อง

2.1.10.2 ข้อจำกัดและการจัดการ

Geolocation API มีข้อจำกัดที่ต้องพิจารณา เช่น ผู้ใช้อาจปฏิเสธการเข้าถึงตำแหน่ง หรือตำแหน่งอาจไม่แม่นยำในบางสภาพแวดล้อม ในแคร์คอนเน็กต์จัดการข้อจำกัดเหล่านี้ด้วยการใช้ Try-Catch เพื่อรองรับกรณีที่ไม่สามารถดึงตำแหน่งได้ และอนุญาตให้ Check-out ได้แม้ไม่มีข้อมูล GPS โดยบังคับให้กรอกหลักฐานการทำงานแทน

### 2.1.11 OTP Verification System

OTP (One-Time Password) เป็นรหัสผ่านที่ใช้ได้ครั้งเดียวและมีอายุการใช้งานจำกัด ซึ่งถูกส่งไปยังผู้ใช้ผ่านช่องทางที่ลงทะเบียนไว้ เช่น อีเมลหรือ SMS เพื่อยืนยันตัวตนและเพิ่มความปลอดภัยของระบบ OTP เป็นส่วนหนึ่งของการยืนยันตัวตนแบบหลายปัจจัย (Multi-Factor Authentication)

2.1.11.1 ขั้นตอนการทำงานของ OTP

1) การร้องขอ OTP: ผู้ใช้กดขอ OTP ระบบจะสร้างรหัสสุ่ม (โดยทั่วไป 6 หลัก) และบันทึกลงในฐานข้อมูลพร้อมกำหนดเวลาหมดอายุ
2) การส่ง OTP: ระบบส่ง OTP ไปยังอีเมลหรือเบอร์โทรศัพท์ของผู้ใช้ผ่าน Provider ที่กำหนด
3) การยืนยัน OTP: ผู้ใช้กรอกรหัส OTP ที่ได้รับ ระบบจะตรวจสอบว่ารหัสถูกต้องและยังไม่หมดอายุ
4) การจำกัดอัตรา (Rate Limiting): ระบบจำกัดจำนวนครั้งในการส่งและยืนยัน OTP เพื่อป้องกันการโจมตีแบบ Brute Force

2.1.11.2 การประยุกต์ใช้ใน CareConnect

ในแคร์คอนเน็กต์ ระบบ OTP ถูกใช้ในการยืนยันอีเมลและเบอร์โทรศัพท์ ซึ่งเป็นเงื่อนไขในการเลื่อน Trust Level จาก L0 ไปเป็น L1 โดยมี API endpoints สำหรับ:
1) ส่ง Email OTP (`POST /api/otp/email/send`)
2) ส่ง Phone OTP (`POST /api/otp/phone/send`)
3) ยืนยัน OTP (`POST /api/otp/verify`)
4) ส่ง OTP ซ้ำ (`POST /api/otp/resend`)

ทุก endpoint ใช้ Rate Limiter เพื่อป้องกันการส่ง OTP มากเกินไป และใช้ requireAuth middleware เพื่อให้เฉพาะผู้ใช้ที่เข้าสู่ระบบแล้วเท่านั้นที่สามารถขอ OTP ได้

### 2.1.12 Responsive Web Design

Responsive Web Design (RWD) เป็นแนวทางการออกแบบเว็บไซต์ที่สามารถปรับการแสดงผลให้เหมาะสมกับขนาดหน้าจอที่แตกต่างกัน ไม่ว่าจะเป็นสมาร์ทโฟน แท็บเล็ต หรือคอมพิวเตอร์ตั้งโต๊ะ โดยใช้เทคนิค CSS Media Queries, Flexible Grid และ Flexible Images

2.1.12.1 หลักการสำคัญ

1) Mobile-first Approach: ออกแบบสำหรับหน้าจอขนาดเล็กก่อน แล้วค่อยขยายสำหรับหน้าจอใหญ่ขึ้น ทำให้มั่นใจว่าประสบการณ์บนมือถือจะดีเสมอ
2) Flexible Layout: ใช้หน่วย Relative เช่น เปอร์เซ็นต์ (%) หรือ rem แทน Pixel เพื่อให้ Layout ยืดหยุ่นตามขนาดหน้าจอ
3) Breakpoints: กำหนดจุดเปลี่ยน Layout ตามขนาดหน้าจอ เช่น sm (640px), md (768px), lg (1024px)

2.1.12.2 การประยุกต์ใช้ใน CareConnect

ในแคร์คอนเน็กต์ใช้ Tailwind CSS ซึ่งรองรับ Responsive Design ด้วย Utility Classes ที่มี Breakpoint Prefixes เช่น `sm:`, `md:`, `lg:` ทำให้สามารถกำหนดรูปแบบที่แตกต่างกันตามขนาดหน้าจอได้โดยไม่ต้องเขียน CSS แยก ระบบถูกออกแบบแบบ Mobile-first โดยเน้นหน้าจอขนาด 360x800 พิกเซล (สมาร์ทโฟนทั่วไป) เป็นหลัก เนื่องจากผู้ใช้ส่วนใหญ่ของแพลตฟอร์มบริการดูแลผู้สูงอายุมักใช้งานผ่านมือถือ องค์ประกอบ UI เช่น Bottom Navigation Bar, Card Layout และ Modal จึงถูกออกแบบให้ใช้งานได้สะดวกบนหน้าจอขนาดเล็ก

## 2.2 ภาพรวมของเครื่องมือที่ใช้ในโครงงาน

โครงงานแคร์คอนเน็กต์ใช้เทคโนโลยีสมัยใหม่ในการพัฒนาเว็บแอปพลิเคชัน โดยแบ่งเป็น 3 ส่วนหลัก คือ Frontend ที่ใช้ React ร่วมกับ TypeScript และ Vite ในการพัฒนาส่วนผู้ใช้งาน Backend ที่ใช้ Node.js ร่วมกับ Express.js ในการจัดการ API และฐานข้อมูล PostgreSQL ในการจัดเก็บข้อมูล นอกจากนี้ยังใช้ Docker Compose ในการจัดการสภาพแวดล้อมการพัฒนา และเครื่องมืออื่น ๆ ที่ช่วยให้การพัฒนามีประสิทธิภาพและเป็นไปอย่างเป็นระบบ

## 2.3 เครื่องมือที่ใช้ในการพัฒนา

### 2.3.1 React

React เป็นไลบรารี JavaScript สำหรับการสร้างส่วนผู้ใช้งาน (User Interface) ที่พัฒนาโดย Facebook ในปี ค.ศ. 2013 React ใช้แนวคิด Component-based Architecture ซึ่งแบ่งส่วนผู้ใช้งานออกเป็นส่วนย่อย ๆ ที่เรียกว่า Component แต่ละ Component จะมีสถานะ (State) และคุณสมบัติ (Props) เป็นของตนเอง ทำให้สามารถนำกลับมาใช้ซ้ำ (Reusable) ได้ React ใช้ Virtual DOM ในการอัปเดตส่วนผู้ใช้งาน ทำให้การแสดงผลมีประสิทธิภาพสูง ในโครงงานแคร์คอนเน็กต์ใช้ React เวอร์ชัน 18 ในการพัฒนาส่วน Frontend ทั้งหมด รวมถึงหน้าแรก หน้าค้นหาผู้ดูแล หน้าจัดการงาน และหน้าแชท

### 2.3.2 TypeScript

TypeScript เป็นภาษาโปรแกรมที่พัฒนาโดย Microsoft ซึ่งเป็น Superset ของ JavaScript โดยเพิ่มความสามารถด้านการกำหนดประเภทข้อมูล (Static Typing) เข้ามา TypeScript ช่วยให้โค้ดมีความชัดเจน ลดข้อผิดพลาดในระยะ runtime และทำให้การบำรุงรักษาโค้ดง่ายขึ้น ในโครงงานแคร์คอนเน็กต์ใช้ TypeScript ในการพัฒนาส่วน Frontend ทั้งหมด โดยกำหนดประเภทข้อมูลสำหรับ props ของ Component ข้อมูลที่ได้จาก API และฟังก์ชันต่าง ๆ ทำให้การพัฒนามีความปลอดภัยและลดข้อผิดพลาดได้มากขึ้น

### 2.3.3 Vite

Vite เป็นเครื่องมือสร้างโปรเจกต์ (Build Tool) สำหรับเว็บแอปพลิเคชันสมัยใหม่ พัฒนาโดย Evan You ผู้สร้าง Vue.js Vite ใช้ Native ES Modules ในการพัฒนา ทำให้การเริ่มต้นเซิร์ฟเวอร์พัฒนามีความเร็วสูง และใช้ Rollup ในการสร้างไฟล์สำหรับใช้งานจริง (Production) ในโครงงานแคร์คอนเน็กต์ใช้ Vite ในการจัดการสภาพแวดล้อมการพัฒนาส่วน Frontend โดยรองรับการพัฒนาแบบ Hot Module Replacement (HMR) ทำให้การแก้ไขโค้ดและดูผลลัพธ์เป็นไปอย่างรวดเร็ว

### 2.3.4 Node.js

Node.js เป็น Runtime Environment สำหรับการรันโค้ด JavaScript นอกเว็บเบราว์เซอร์ พัฒนาโดย Ryan Dahl ในปี ค.ศ. 2009 Node.js ใช้ V8 JavaScript Engine จาก Google Chrome และรองรับการทำงานแบบ Asynchronous I/O ด้วย Event Loop ทำให้สามารถจัดการการเชื่อมต่อจำนวนมากได้อย่างมีประสิทธิภาพ ในโครงงานแคร์คอนเน็กต์ใช้ Node.js เวอร์ชัน 18 ในการพัฒนาส่วน Backend ทั้งหมด โดยใช้ในการรันเซิร์ฟเวอร์ API จัดการการเชื่อมต่อฐานข้อมูล และประมวลผลตรรกะทางธุรกิจต่าง ๆ

### 2.3.5 Express.js

Express.js เป็น Web Framework สำหรับ Node.js ที่เป็น Minimalist และ Flexible พัฒนาโดย TJ Holowaychuk Express.js ให้ความสามารถในการจัดการ HTTP Request/Response การจัดการ Routing และ Middleware ทำให้การพัฒนา Web API มีความสะดวกและรวดเร็ว ในโครงงานแคร์คอนเน็กต์ใช้ Express.js เวอร์ชัน 4 ในการพัฒนา RESTful API ทั้งหมด โดยแบ่งเป็น Routes ตามฟังก์ชันการทำงาน เช่น Authentication, Profile, Jobs, Wallet, Chat และ Dispute

### 2.3.6 PostgreSQL

PostgreSQL เป็นระบบจัดการฐานข้อมูลเชิงสัมพันธ์ (Relational Database Management System) ที่เป็น Open Source พัฒนาโดย University of California, Berkeley เป็นฐานข้อมูลที่มีความน่าเชื่อถือสูง รองรับ ACID Transactions และมีความสามารถในการจัดการข้อมูลขนาดใหญ่ PostgreSQL รองรับประเภทข้อมูลที่หลากหลาย เช่น JSON, Array และ Geographic Data ในโครงงานแคร์คอนเน็กต์ใช้ PostgreSQL เวอร์ชัน 15 ในการจัดเก็บข้อมูลทั้งหมด รวมถึงข้อมูลผู้ใช้งาน ข้อมูลงาน ข้อมูลการทำธุรกรรม และข้อมูลแชท โดยมีการออกแบบ Database Schema ที่ประกอบด้วย 40 tables

### 2.3.7 Docker

Docker เป็นแพลตฟอร์มสำหรับการสร้างและจัดการ Containers พัฒนาโดย Solomon Hykes Docker ใช้ Linux Containers (LXC) ในการแยกสภาพแวดล้อมการทำงานของแอปพลิเคชัน ทำให้สามารถรันแอปพลิเคชันได้อย่างสม่ำเสมอในทุกสภาพแวดล้อม ในโครงงานแคร์คอนเน็กต์ใช้ Docker ในการจัดการสภาพแวดล้อมการพัฒนา โดยใช้ Docker Compose ในการจัดการหลาย Services พร้อมกัน รวมถึง PostgreSQL, Backend API, Frontend และ Mock Provider

### 2.3.8 Docker Compose

Docker Compose เป็นเครื่องมือสำหรับการจัดการหลาย Containers พร้อมกัน โดยใช้ YAML file ในการกำหนดค่าของแต่ละ Service Docker Compose ช่วยให้สามารถสร้าง รัน และจัดการหลาย Containers ได้อย่างง่าย ในโครงงานแคร์คอนเน็กต์ใช้ Docker Compose ในการจัดการสภาพแวดล้อมการพัฒนาทั้งหมด โดยกำหนด Services สำหรับ PostgreSQL, Backend API, Frontend, Mock Provider และ pgAdmin ทำให้สามารถเริ่มต้นสภาพแวดล้อมการพัฒนาได้ด้วยคำสั่งเดียว

### 2.3.9 Tailwind CSS

Tailwind CSS เป็น CSS Framework แบบ Utility-first พัฒนาโดย Adam Wathan Tailwind CSS ให้คลาสยูทิลิตี้ต่าง ๆ สำหรับการจัดรูปแบบส่วนผู้ใช้งาน ทำให้สามารถสร้างการออกแบบที่ไม่ซ้ำกันได้อย่างรวดเร็ว โดยไม่ต้องเขียน CSS แบบดั้งเดิม ในโครงงานแคร์คอนเน็กต์ใช้ Tailwind CSS เวอร์ชัน 3 ในการจัดรูปแบบส่วนผู้ใช้งานทั้งหมด โดยใช้คลาสยูทิลิตี้ในการจัดรูปแบบ Layout, Typography, Colors และ Responsive Design

### 2.3.10 Socket.IO

Socket.IO เป็นไลบรารี JavaScript สำหรับการสื่อสารแบบ Real-time ระหว่าง Client และ Server โดยใช้ WebSocket หรือ HTTP Long Polling เป็นพื้นฐาน Socket.IO ให้ความสามารถในการส่งข้อมูลแบบ Bi-directional และมีการจัดการการเชื่อมต่ออัตโนมัติ ในโครงงานแคร์คอนเน็กต์ใช้ Socket.IO เวอร์ชัน 4 ในการพัฒนาระบบแชทเรียลไทม์และระบบแจ้งเตือน โดยทำให้ผู้ใช้งานสามารถส่งข้อความและรับการแจ้งเตือนได้ทันที

### 2.3.11 JWT (JSON Web Token)

JWT เป็นมาตรฐาน (RFC 7519) สำหรับการสร้าง Token ในการยืนยันตัวตนแบบ Stateless JWT ประกอบด้วย Header, Payload และ Signature ที่ถูกเข้ารหัสด้วย Base64 ในโครงงานแคร์คอนเน็กต์ใช้ JWT ในการยืนยันตัวตนผู้ใช้งานทุกครั้งที่เข้าใช้งานระบบ โดยใช้ Access Token สำหรับการเข้าถึงระบบในระยะสั้น และ Refresh Token สำหรับการต่ออายุการใช้งานในระยะยาว

### 2.3.12 bcrypt

bcrypt เป็นฟังก์ชันแฮชรหัสผ่านที่ออกแบบมาเพื่อความปลอดภัยสูง bcrypt ใช้ Salt ในการเข้ารหัสรหัสผ่าน ทำให้การถอดรหัสยากและช้า ในโครงงานแคร์คอนเน็กต์ใช้ bcrypt ในการเข้ารหัสรหัสผ่านผู้ใช้งานก่อนจัดเก็บในฐานข้อมูล ทำให้รหัสผ่านมีความปลอดภัยสูงแม้กรณีที่ฐานข้อมูลถูกรั่วไหล

### 2.3.13 Joi

Joi เป็นไลบรารีสำหรับการตรวจสอบความถูกต้องของข้อมูล (Data Validation) สำหรับ JavaScript ที่พัฒนาโดย Walmart Labs Joi ให้ความสามารถในการกำหนด Schema สำหรับตรวจสอบข้อมูล โดยรองรับประเภทข้อมูลหลากหลายและมี Error Messages ที่ชัดเจน ในโครงงานแคร์คอนเน็กต์ใช้ Joi ในการตรวจสอบความถูกต้องของข้อมูลที่รับมาจากผู้ใช้งานทุก API endpoint ทำให้ระบบมีความปลอดภัยและลดข้อผิดพลาดจากข้อมูลที่ไม่ถูกต้อง

### 2.3.14 pg (PostgreSQL Client for Node.js)

pg เป็นไลบรารี Node.js สำหรับการเชื่อมต่อกับฐานข้อมูล PostgreSQL pg ให้ความสามารถในการดำเนินการ SQL queries และจัดการ Connection Pool ในโครงงานแคร์คอนเน็กต์ใช้ pg ในการเชื่อมต่อฐานข้อมูล PostgreSQL จากส่วน Backend โดยใช้ในการดำเนินการ CRUD operations และจัดการธุรกรรมต่าง ๆ

### 2.3.15 Axios

Axios เป็นไลบรารี HTTP Client สำหรับ JavaScript ที่ใช้ Promise-based API Axios ให้ความสามารถในการส่ง HTTP requests และจัดการ responses โดยรองรับการทำงานแบบ Asynchronous ในโครงงานแคร์คอนเน็กต์ใช้ Axios ในส่วน Frontend สำหรับการสื่อสารกับ Backend API โดยใช้ในการดึงข้อมูล ส่งข้อมูล และจัดการการยืนยันตัวตน

### 2.3.16 React Hot Toast

React Hot Toast เป็นไลบรารีสำหรับการแสดง Notifications ใน React ที่มีความเรียบง่ายและใช้งานง่าย React Hot Toast ให้ความสามารถในการแสดงข้อความแจ้งเตือนประเภทต่าง ๆ เช่น Success, Error, Loading ในโครงงานแคร์คอนเน็กต์ใช้ React Hot Toast ในการแสดงข้อความแจ้งเตือนต่าง ๆ ให้กับผู้ใช้งาน เช่น การลงทะเบียนสำเร็จ การสร้างงานสำเร็จ และข้อผิดพลาดต่าง ๆ

### 2.3.17 Lucide Icons

Lucide Icons เป็นชุดไอคอนที่มีความสวยงามและสม่ำเสมอ พัฒนาโดยผู้สร้าง Feather Icons Lucide Icons ให้ไอคอนที่มีความละเอียดสูงและรองรับหลายขนาด ในโครงงานแคร์คอนเน็กต์ใช้ Lucide Icons ในการแสดงไอคอนต่าง ๆ ในส่วนผู้ใช้งาน เช่น ไอคอนเมนู ไอคอนฟังก์ชัน และไอคอนสถานะ

### 2.3.18 Playwright

Playwright เป็นเครื่องมือสำหรับการทดสอบแอปพลิเคชันแบบ End-to-End (E2E) ที่พัฒนาโดย Microsoft Playwright ให้ความสามารถในการทดสอบแอปพลิเคชันบนเว็บเบราว์เซอร์ต่าง ๆ เช่น Chromium, Firefox และ WebKit ในโครงงานแคร์คอนเน็กต์วางแผนที่จะใช้ Playwright ในการทดสอบฟังก์ชันการทำงานสำคัญของระบบ เช่น การลงทะเบียน การเข้าสู่ระบบ การสร้างงาน และการชำระเงิน

การพัฒนาเว็บแอปพลิเคชันแคร์คอนเน็กต์จำเป็นต้องทบทวนงานวิจัยและแอปพลิเคชันที่เกี่ยวข้อง เพื่อศึกษาแนวทางการพัฒนาแพลตฟอร์มบริการดูแลผู้สูงอายุ ระบบตลาดสองด้าน ระบบความน่าเชื่อถือ และระบบชำระเงินที่ปลอดภัย ซึ่งสามารถสรุปได้ดังต่อไปนี้

**1) สถานการณ์สังคมผู้สูงอายุในประเทศไทย**

ประเทศไทยก้าวเข้าสู่สังคมสูงอายุอย่างสมบูรณ์ (Complete Aged Society) แล้วในปี พ.ศ. 2567 โดยมีประชากรอายุ 60 ปีขึ้นไปจำนวน 13.2 ล้านคน คิดเป็นสัดส่วนร้อยละ 20 ของประชากรทั้งหมด (สำนักงานสภาพัฒนาการเศรษฐกิจและสังคมแห่งชาติ, 2567) สถานการณ์ดังกล่าวส่งผลให้ความต้องการผู้ดูแลผู้สูงอายุเพิ่มสูงขึ้นอย่างต่อเนื่อง แต่การจ้างงานผู้ดูแลในปัจจุบันยังขาดแพลตฟอร์มกลางที่มีมาตรฐาน ส่วนใหญ่ยังอาศัยช่องทางที่ไม่เป็นทางการ เช่น โซเชียลมีเดียหรือการแนะนำปากต่อปาก ทำให้เกิดปัญหาด้านความโปร่งใสและความปลอดภัย การพัฒนาแพลตฟอร์มที่มีระบบตรวจสอบตัวตน ระบบความน่าเชื่อถือ และระบบชำระเงินที่ปลอดภัย จึงเป็นแนวทางที่ตอบโจทย์ความต้องการของสังคมไทยในปัจจุบัน

**2) แพลตฟอร์มบริการดูแลผู้สูงอายุที่เกี่ยวข้อง**

ปัจจุบันมีแพลตฟอร์มบริการดูแลผู้สูงอายุหลายแห่งที่ประสบความสำเร็จในต่างประเทศ ซึ่งสามารถนำมาศึกษาเป็นแนวทางในการพัฒนาแคร์คอนเน็กต์ได้ ดังนี้

**Care.com** เป็นแพลตฟอร์มตลาดสองด้าน (Two-sided Marketplace) ที่ใหญ่ที่สุดในสหรัฐอเมริกา ก่อตั้งในปี ค.ศ. 2007 มีสมาชิกมากกว่า 16.5 ล้านคน ให้บริการจับคู่ครอบครัวกับผู้ดูแลหลากหลายประเภท โดยมีระบบตรวจสอบประวัติอาชญากรรม (Background Check) ระบบรีวิวจากผู้ใช้งาน และระบบสมัครสมาชิกแบบรายเดือน Care.com ได้รับประโยชน์จาก indirect network effects กล่าวคือ ยิ่งมีผู้ดูแลและครอบครัวบนแพลตฟอร์มมากขึ้น ก็ยิ่งเพิ่มทางเลือกและความยืดหยุ่นให้กับทั้งสองฝ่าย (Harvard Business School Digital Initiative, 2016)

**Honor** เป็นแพลตฟอร์มบริการดูแลผู้สูงอายุที่บ้านในสหรัฐอเมริกา ก่อตั้งในปี ค.ศ. 2014 ใช้เทคโนโลยีในการจับคู่ผู้ดูแลกับผู้รับบริการ โดยพิจารณาจากทักษะ ประสบการณ์ พื้นที่ และตารางเวลา Honor มีแอปพลิเคชันสำหรับผู้ดูแลที่ช่วยในการจัดการตารางงาน บันทึกการดูแล และรับการฝึกอบรม ในปี ค.ศ. 2021 Honor ได้ควบรวมกิจการกับ Home Instead ซึ่งเป็นเครือข่ายบริการดูแลผู้สูงอายุที่ใหญ่ที่สุดในโลก ทำให้กลายเป็นเครือข่ายดูแลผู้สูงอายุที่บ้านที่ใหญ่ที่สุด (TechCrunch, 2021)

**Homage** เป็นแพลตฟอร์มบริการดูแลสุขภาพที่บ้านในภูมิภาคเอเชียตะวันออกเฉียงใต้ ก่อตั้งในประเทศสิงคโปร์เมื่อปี ค.ศ. 2016 มีเครือข่ายผู้ดูแลที่ผ่านการคัดกรองมากกว่า 2,000 คน Homage ให้บริการจับคู่ผู้ดูแลกับผู้รับบริการ โดยรองรับทั้งการดูแลผู้สูงอายุ การพยาบาล และการกายภาพบำบัด แพลตฟอร์มนี้ใช้ระบบจับคู่อัตโนมัติที่พิจารณาจากความต้องการของผู้รับบริการ ทักษะของผู้ดูแล และพื้นที่ให้บริการ (Forbes, 2022)

**CareLinx** เป็นแพลตฟอร์มจับคู่ผู้ดูแลกับครอบครัวในสหรัฐอเมริกา มีเครือข่ายผู้ดูแลมากกว่า 450,000 คน CareLinx มีระบบตรวจสอบประวัติ ระบบรีวิว การสื่อสารแบบเรียลไทม์ และระบบจัดการการชำระเงิน โดยได้รับความไว้วางใจจากบริษัทประกันสุขภาพและนายจ้างรายใหญ่ในการให้บริการดูแลผู้สูงอายุ (CareLinx, 2024)

จากตารางเปรียบเทียบจะเห็นได้ว่า แคร์คอนเน็กต์มีฟีเจอร์ที่ครอบคลุมกว่าแพลตฟอร์มที่มีอยู่ โดยเฉพาะระบบระดับความน่าเชื่อถือ (Trust Level) ที่แบ่งเป็น 4 ระดับ ระบบชำระเงินแบบพักเงิน (Escrow) ที่ป้องกันทั้งผู้ว่าจ้างและผู้ดูแล ระบบ Check-in/Check-out ด้วย GPS และระบบจัดการข้อพิพาท นอกจากนี้แคร์คอนเน็กต์ยังเป็นแพลตฟอร์มที่ออกแบบมาสำหรับบริบทของประเทศไทยโดยเฉพาะ

**3) ระบบตลาดสองด้าน (Two-Sided Marketplace)**

แนวคิดตลาดสองด้าน (Two-Sided Marketplace) เป็นรูปแบบธุรกิจแพลตฟอร์มที่เชื่อมต่อผู้ใช้งานสองกลุ่มเข้าด้วยกัน โดยสร้างมูลค่าจาก indirect network effects กล่าวคือ เมื่อจำนวนผู้ใช้งานฝั่งหนึ่งเพิ่มขึ้น จะดึงดูดให้ผู้ใช้งานอีกฝั่งเข้ามาใช้แพลตฟอร์มมากขึ้นตามไปด้วย (Parker, Van Alstyne & Choudary, 2016) แพลตฟอร์มบริการดูแลผู้สูงอายุ เช่น Care.com ใช้รูปแบบนี้ในการเชื่อมต่อครอบครัวกับผู้ดูแล โดยผู้ว่าจ้างจะมีทางเลือกในการหาผู้ดูแลมากขึ้นเมื่อมีผู้ดูแลบนแพลตฟอร์มจำนวนมาก และในทางกลับกัน ผู้ดูแลก็จะมีโอกาสได้รับงานมากขึ้นเมื่อมีผู้ว่าจ้างจำนวนมาก งานวิจัยของ Tse et al. (2020) ชี้ว่าระบบจับคู่ที่มีประสิทธิภาพในแพลตฟอร์มบริการดูแลควรพิจารณาปัจจัยด้านพื้นที่ เวลา ประสบการณ์ และประเภทบริการ แคร์คอนเน็กต์ใช้แนวคิดนี้ในการออกแบบระบบประกาศงานและค้นหาผู้ดูแล โดยรองรับการกรองตามทักษะ ใบรับรอง พื้นที่ และตารางเวลาว่าง

**4) ระบบความน่าเชื่อถือและการยืนยันตัวตนในแพลตฟอร์มออนไลน์**

ความไว้วางใจ (Trust) เป็นปัจจัยสำคัญที่สุดในแพลตฟอร์มบริการที่ผู้ใช้งานต้องพบกันในสถานที่จริง โดยเฉพาะบริการดูแลสุขภาพที่ต้องมีความน่าเชื่อถือสูง งานวิจัยของ Raimo et al. (2023) ระบุว่าการยืนยันตัวตนผ่านเอกสารราชการ การตรวจสอบประวัติ และระบบรีวิวหลังการให้บริการ เป็นกลไกที่ช่วยเสริมความปลอดภัยในแพลตฟอร์มสุขภาพออนไลน์ นอกจากนี้ระบบ Know Your Customer (KYC) ซึ่งเป็นกระบวนการยืนยันตัวตนที่ใช้กันอย่างแพร่หลายในภาคการเงิน ได้ถูกนำมาประยุกต์ใช้กับแพลตฟอร์มบริการเพื่อสร้างความน่าเชื่อถือ โดยผู้ใช้ต้องส่งเอกสารยืนยันตัวตน เช่น บัตรประชาชน และรูปถ่ายใบหน้า (Selfie) เพื่อเปรียบเทียบกับเอกสาร แคร์คอนเน็กต์นำแนวคิดเหล่านี้มาใช้ในการออกแบบระบบระดับความน่าเชื่อถือ (Trust Level) 4 ระดับ ตั้งแต่ L0 (ยังไม่ยืนยัน) ไปจนถึง L3 (เชื่อถือได้สูงสุด) โดยผู้ใช้ต้องผ่านการยืนยัน OTP, KYC และสะสมคะแนนความน่าเชื่อถือจากการใช้งานจริง

**5) ระบบชำระเงินแบบพักเงิน (Escrow) ในเศรษฐกิจแบบ Gig**

ระบบชำระเงินแบบพักเงิน (Escrow) เป็นกลไกที่บุคคลที่สามทำหน้าที่เก็บรักษาเงินไว้จนกว่าทั้งสองฝ่ายจะปฏิบัติตามเงื่อนไขที่ตกลงกัน ซึ่งช่วยลดความเสี่ยงของทั้งผู้ว่าจ้างและผู้ให้บริการ ในเศรษฐกิจแบบ Gig ระบบ Escrow ถูกนำมาใช้อย่างแพร่หลายในแพลตฟอร์ม เช่น Upwork และ Freelancer.com เพื่อสร้างความมั่นใจว่าผู้ให้บริการจะได้รับค่าตอบแทนเมื่อทำงานเสร็จ และผู้ว่าจ้างจะได้รับงานตามที่ตกลง (Castler, 2024) แคร์คอนเน็กต์นำระบบ Escrow มาประยุกต์ใช้กับบริการดูแลผู้สูงอายุ โดยเงินจะถูกพักไว้ในกระเป๋าเงิน Escrow เมื่อผู้ดูแลรับงาน และจะถูกโอนให้ผู้ดูแลเมื่อทำงานเสร็จสิ้นและ Check-out เรียบร้อยแล้ว ซึ่งช่วยป้องกันปัญหาการไม่ชำระเงินและสร้างความมั่นใจให้กับทั้งสองฝ่าย

จากการทบทวนงานวิจัยและแอปพลิเคชันที่เกี่ยวข้องเหล่านี้ จะเห็นได้ว่าการพัฒนาแพลตฟอร์มบริการดูแลผู้สูงอายุต้องคำนึงถึงทั้งระบบจับคู่ที่มีประสิทธิภาพ ระบบความน่าเชื่อถือที่แข็งแกร่ง และระบบชำระเงินที่ปลอดภัย แคร์คอนเน็กต์ได้นำแนวคิดและบทเรียนจากแพลตฟอร์มเหล่านี้มาประยุกต์ใช้ พร้อมเพิ่มฟีเจอร์ที่ยังไม่มีในแพลตฟอร์มที่มีอยู่ เช่น ระบบระดับความน่าเชื่อถือ 4 ระดับ ระบบ Escrow สำหรับบริการดูแล ระบบ Check-in/Check-out ด้วย GPS และระบบจัดการข้อพิพาท เพื่อให้ตอบโจทย์บริบทของประเทศไทยที่กำลังเข้าสู่สังคมสูงอายุอย่างสมบูรณ์

## 2.5 กฎหมายและข้อบังคับที่เกี่ยวข้อง

การพัฒนาแพลตฟอร์มบริการดูแลผู้สูงอายุในประเทศไทยจำเป็นต้องปฏิบัติตามกฎหมายและข้อบังคับที่เกี่ยวข้องหลายฉบับ โดยเฉพาะด้านการคุ้มครองผู้สูงอายุ การคุ้มครองข้อมูลส่วนบุคคล การจ้างงาน และการให้บริการทางอิเล็กทรอนิกส์ ซึ่งสามารถแบ่งได้ดังนี้

### 2.5.1 กฎหมายด้านการคุ้มครองผู้สูงอายุ

**พระราชบัญญัติผู้สูงอายุ พ.ศ. 2546**

1) รายละเอียด: เป็นกฎหมายหลักที่คุ้มครองสิทธิและผลประโยชน์ของผู้สูงอายุในประเทศไทย กำหนดหน้าที่ของรัฐและสังคมในการดูแลผู้สูงอายุ
2) ข้อกำหนดที่เกี่ยวข้อง:
• รัฐต้องส่งเสริมและสนับสนุนการดูแลผู้สูงอายุให้เป็นไปอย่างมีคุณภาพ
• การจัดให้มีบริการดูแลผู้สูงอายุที่เข้าถึงได้และเป็นธรรม
• การคุ้มครองผู้สูงอายุจากการถูกทอดทิ้ง ละเลย หรือถูกกระทำความรุนแรง
• การส่งเสริมให้ภาคเอกชนมีส่วนร่วมในการดูแลผู้สูงอายุ

### 2.5.2 กฎหมายด้านการคุ้มครองข้อมูลส่วนบุคคล

**พระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA)**

1) รายละเอียด: คุ้มครองข้อมูลส่วนบุคคลของผู้ใช้บริการ ป้องกันการนำข้อมูลไปใช้โดยไม่ได้รับอนุญาต ใช้บังคับกับทุกภาคส่วนที่เก็บรวบรวมข้อมูลส่วนบุคคลในประเทศไทย
2) ข้อกำหนดที่เกี่ยวข้อง:
• ต้องได้รับความยินยอมจากเจ้าของข้อมูลก่อนการเก็บ รวบรวม หรือใช้ข้อมูล
• ต้องเก็บรักษาข้อมูลส่วนบุคคลอย่างปลอดภัยด้วยมาตรการที่เหมาะสม
• ต้องเปิดเผยวัตถุประสงค์ในการใช้ข้อมูลอย่างชัดเจน
• เจ้าของข้อมูลมีสิทธิในการเข้าถึง แก้ไข หรือลบข้อมูลของตนเอง
• ต้องแต่งตั้งผู้คุ้มครองข้อมูลส่วนบุคคล (DPO) สำหรับองค์กรที่มีการประมวลผลข้อมูลขนาดใหญ่

### 2.5.3 กฎหมายด้านการจ้างงานและแรงงาน

**พระราชบัญญัติคุ้มครองแรงงาน พ.ศ. 2541**

1) รายละเอียด: กำหนดสิทธิและหน้าที่ของนายจ้างและลูกจ้าง รวมถึงสวัสดิการและความปลอดภัยในการทำงาน ใช้บังคับกับทุกความสัมพันธ์ระหว่างนายจ้างและลูกจ้าง
2) ข้อกำหนดที่เกี่ยวข้อง:
• ต้องทำสัญญาจ้างงานที่ชัดเจนและเป็นธรรมกับผู้ดูแล
• ต้องจ่ายค่าจ้างตามที่ตกลงกันและไม่ต่ำกว่าค่าจ้างขั้นต่ำ
• ต้องจัดสวัสดิการและค่าตอบแทนที่เหมาะสมตามกฎหมาย
• ต้องปฏิบัติตามมาตรฐานความปลอดภัยและสุขอนามัยในการทำงาน

**กฎหมายประกันสังคม**

1) รายละเอียด: ให้ความคุ้มครองทางสังคมแก่ลูกจ้างในกรณีต่าง ๆ เช่น การเจ็บป่วย อุบัติเหตุ และการว่างงาน บังคับใช้กับนายจ้างที่มีพนักงานตั้งแต่ 1 คนขึ้นไป
2) ข้อกำหนดที่เกี่ยวข้อง:
• การขึ้นทะเบียนประกันสังคมสำหรับผู้ดูแลที่เป็นพนักงาน
• การส่งเงินสมทบประกันสังคมตามที่กฎหมายกำหนด (อัตรา 5% ของค่าจ้าง)
• การให้สิทธิประโยชน์ต่าง ๆ แก่ผู้ดูแลตามกฎหมาย เช่น การเจ็บป่วย การคลอดบุตร การว่างงาน

### 2.5.4 กฎหมายด้านการให้บริการทางอิเล็กทรอนิกส์

**พระราชบัญญัติธุรกรรมทางอิเล็กทรอนิกส์ พ.ศ. 2544**

1) รายละเอียด: ควบคุมการทำธุรกรรมและการสื่อสารผ่านระบบอิเล็กทรอนิกส์ให้ปลอดภัยและเชื่อถือได้ ให้ความสำคัญกับการรับรองความถูกต้องของข้อมูลอิเล็กทรอนิกส์
2) ข้อกำหนดที่เกี่ยวข้อง:
• การรักษาความปลอดภัยของข้อมูลที่ส่งผ่านระบบ
• การยืนยันตัวตนและความถูกต้องของข้อมูลที่เผยแพร่
• การปฏิบัติตามมาตรฐานด้านความปลอดภัยไซเบอร์
• การให้ความสำคัญกับการเก็บรักษาหลักฐานทางอิเล็กทรอนิกส์

**พระราชบัญญัติการกระทำผิดเกี่ยวกับคอมพิวเตอร์ พ.ศ. 2550 (Cybersecurity Act)**

1) รายละเอียด: ป้องกันและปราบปรามการกระทำผิดเกี่ยวกับคอมพิวเตอร์และการใช้บริการคอมพิวเตอร์ผิดกฎหมาย
2) ข้อกำหนดที่เกี่ยวข้อง:
• ต้องมีมาตรการป้องกันการเข้าถึงข้อมูลโดยไม่ได้รับอนุญาต
• ต้องรักษาความปลอดภัยของระบบคอมพิวเตอร์อย่างเข้มแข็ง
• ต้องรายงานเหตุการณ์ความไม่ปลอดภัยทางไซเบอร์ให้ทราบต่อหน่วยงานที่เกี่ยวข้อง

### 2.5.5 กฎหมายด้านการประกอบวิชาชีพดูแลสุขภาพ

**พระราชบัญญัติวิชาชีพผู้ดูแลสุขภาพ พ.ศ. 2556**

1) รายละเอียด: ควบคุมการประกอบวิชาชีพผู้ดูแลสุขภาพ เพื่อให้การดูแลมีมาตรฐานและความปลอดภัย
2) ข้อกำหนดที่เกี่ยวข้อง:
• ผู้ดูแลสุขภาพต้องมีใบประกอบวิชาชีพที่ถูกต้อง
• ต้องปฏิบัติตามจรรยาบรรณและมาตรฐานวิชาชีพ
• ต้องมีการพัฒนาความรู้และทักษะอย่างต่อเนื่อง

**กฎหมายว่าด้วยการปฏิบัติงานของบุคลากรด้านสาธารณสุข**

1) รายละเอียด: กำหนดมาตรฐานและข้อบังคับสำหรับบุคลากรด้านสาธารณสุขที่ไม่ใช่แพทย์และพยาบาล
2) ข้อกำหนดที่เกี่ยวข้อง:
• ต้องผ่านการอบรมตามหลักสูตรที่กระทรวงสาธารณสุขกำหนด
• ต้องมีใบรับรองการอบรมที่ยังไม่หมดอายุ
• ต้องปฏิบัติตามข้อบังคับของสถานพยาบาลและสถานที่ให้บริการ

### 2.5.6 กฎหมายด้านการคุ้มครองผู้บริโภค

**พระราชบัญญัติคุ้มครองผู้บริโภค พ.ศ. 2522**

1) รายละเอียด: ปกป้องสิทธิของผู้บริโภคในการรับสินค้าและบริการที่มีคุณภาพและปลอดภัย ใช้บังคับกับทุกผู้ประกอบธุรกิจ
2) ข้อกำหนดที่เกี่ยวข้อง:
• ต้องให้ข้อมูลที่ถูกต้องและครบถ้วนเกี่ยวกับบริการดูแลผู้สูงอายุ
• ต้องรับผิดชอบต่อความเสียหายที่เกิดจากการให้บริการ
• ต้องมีช่องทางให้ผู้ว่าจ้างร้องเรียนหรือเสนอข้อเสนอแนะได้
• ห้ามโฆษณาหรือให้ข้อมูลที่หลอกลวงเกี่ยวกับคุณสมบัติของผู้ดูแล

### 2.5.7 กฎหมายด้านภาษีและการเงิน

**พระราชบัญญัติภาษีมูลค่าเพิ่ม พ.ศ. 2534**

1) รายละเอียด: กำหนดการเก็บภาษีจากการขายสินค้าและบริการ บังคับใช้กับผู้ประกอบการที่มีรายได้ต่อปีเกินกว่าที่กฎหมายกำหนด
2) ข้อกำหนดที่เกี่ยวข้อง:
• การจดทะเบียนภาษีมูลค่าเพิ่มสำหรับธุรกิจแพลตฟอร์ม
• การออกใบเสร็จรับเงินและใบกำกับภาษีที่ถูกต้อง
• การยื่นและชำระภาษีตามระยะเวลาที่กฎหมายกำหนด

**พระราชบัญญัติป้องกันและปราบปรามการฟอกเงิน พ.ศ. 2542**

1) รายละเอียด: ป้องกันการใช้ระบบการเงินในการฟอกเงินและกิจกรรมที่ผิดกฎหมาย ใช้บังคับกับสถาบันการเงินและผู้ประกอบการที่เกี่ยวข้องกับการเงิน
2) ข้อกำหนดที่เกี่ยวข้อง:
• การตรวจสอบและยืนยันตัวตนของผู้ใช้บริการ (KYC)
• การรายงานธุรกรรมที่น่าสงสัยต่อหน่วยงานที่เกี่ยวข้อง
• การเก็บรักษาข้อมูลธุรกรรมตามระยะเวลาที่กฎหมายกำหนด
• การจัดทำนโยบายต้านการฟอกเงิน (AML Policy)

จากกฎหมายและข้อบังคับที่เกี่ยวข้องเหล่านี้ แคร์คอนเน็กต์จำเป็นต้องออกแบบระบบให้สอดคล้องกับกฎหมายไทย โดยเฉพาะการคุ้มครองข้อมูลส่วนบุคคลตาม PDPA การปฏิบัติตามกฎหมายแรงงานสำหรับผู้ดูแล และการจัดการธุรกรรมทางการเงินตามมาตรฐานป้องกันการฟอกเงิน ซึ่งส่งผลให้ระบบมีความปลอดภัย เชื่อถือได้ และสอดคล้องกับกฎหมายประเทศไทย
